    神奇，当v4=0的时候，是子进程，v4!=0 就是父进程

    wait ()会暂时停止目前进程的执行, 直到有信号来到或子进程结束

    经常调着调着程序停了¶

    在开头定了个闹钟，改大一点就行

    是一个虚拟机

    地址 v8 作用 对应
    401780h 3 未知
    qword0 = 0 0 跳转base到q2 nop
    #### case 2 和 case 4 是一个相反的操作

    2是：ret 跳转到 base = v13[...] v12 += 8

    pop loc1
jmp loc1

    4是：call v13[...] = base->q0 ， __iptr++ v12 -= 8 base = base-> q1

    push loc1
jmp loc2

    当 pfunc0 = 0 或 v8 = 0¶

    都像是jmp

    pfunc0 == 0 -> jmp base->q1
v8 == 0 -> jmp base->q0
v8 == 1 -> jmp base->q1

    ptrace(PTRACE_GETREGS, a1, 0LL, v10);

    PTRACE_PEEKDATA

    大概知道了，有以下难点：¶

    1. 控制流平坦化——由父进程统一进行代码流程分发
    2. 虚拟机分析——jmp、call、push、pop均由父进程操作
    3. 多线程交互——子进程的部分栈操作，用父进程模拟

    目前想到的方案是用frida hook每一个流程，然后把程序的流程用正常的汇编表示
